# Android Native vs Web Mobile Parity Audit

Scope: Android native `staging/prod` — zero web fallback policy. All screens and data are fully native Compose + Convex. Mobile web is the source of truth for feature parity but is never loaded at runtime.

Baseline sources:

- Web route inventory: `src/routes.tsx`
- Web mobile chrome and bottom-nav semantics: `src/config/routes.config.ts`, `src/components/mobile/MobileBottomNav.tsx`
- Android native route inventory: `apps/mobile/android/app/src/main/java/com/hangyeol/app/compose/navigation/HangyeolRouteBaseline.kt`
- Android native shell: `apps/mobile/android/app/src/main/java/com/hangyeol/app/compose/ui/HangyeolComposeApp.kt`
- Android parity screens: `apps/mobile/android/app/src/main/java/com/hangyeol/app/compose/ui/ParityScreens.kt`

## Zero Web Fallback Policy

All `NativeParityFallbackScreen`, `WebFallbackScreen`, `HangyeolDestination.WebFallback`, `buildWebUrl`, and `openWebPath` have been removed. Unrecognized routes resolve to `TabsToday` (Dashboard). No web URLs are opened from the Android app at runtime for feature delivery.

## Hardcoded Data Policy

- `staging/prod` Android repositories use only Convex data. If Convex fails, screens show cached data, error state, or empty state. No sample people, sample podcasts, fake video subtitles, fake rankings, or fake challenge data are ever rendered.
- `mock` flavor still uses local `DataStore*`/in-memory repositories for preview and instrumentation. This is intentionally excluded from the parity scope.
- Static UI labels, option labels, and section titles remain local UI copy.

## Web Fallback Status

**ELIMINATED.** The `WebFallback` destination, `NativeParityFallbackScreen`, `WebFallbackScreen`, and all web URL helpers (`buildWebUrl`, `openWebPath`, `WEB_APP_BASE_URL`) have been removed from the codebase. All routes resolve to native screens. Unrecognized paths default to Dashboard (Today).
